الإضافات

الـ plugins هي أعلى مستوى من آليات التوسعة في Claude Code. تجمع skills و subagents و hooks و MCP servers وإعدادات LSP في حزمة واحدة قابلة للتثبيت. الفريق يثبّت plugin واحد وكل شيء يتضبط فورًا — لا setup يدوي لكل مكوّن. هذه الوحدة تغطي هيكل الـ plugin، صيغة الـ manifest، طرق التوزيع، وكيف تبني الـ plugin بنفسك.

هيكل الـ Plugin

الـ plugin هو مجلد بهيكل محدد. الملف الوحيد المطلوب هو .claude-plugin/plugin.json، الـ manifest الذي يعرّف هوية الـ plugin. كل شيء آخر اختياري لكنه يتبع conventions تتعرف عليها Claude Code:

my-plugin/ (هيكل المجلد)
my-plugin/
├── .claude-plugin/
│   └── plugin.json       # الـ manifest المطلوب
├── skills/               # ملفات SKILL.md
│   └── my-skill/
│       └── SKILL.md
├── agents/               # تعريفات الـ subagents
│   └── specialist.md
├── commands/             # ملفات الأوامر القديمة (تعمل كذلك)
│   └── my-command.md
├── hooks/
│   └── hooks.json        # hooks على مستوى الـ plugin
├── .mcp.json             # إعدادات خوادم MCP
├── .lsp.json             # إعدادات خوادم LSP
├── settings.json         # الإعدادات الافتراضية
└── bin/
    └── helper.sh

الـ plugin الذي يشحن skill واحدة فقط يمكنه وضع SKILL.md مباشرة في root الـ plugin بدلًا من إنشاء مجلد skills/. تحمّلها Claude Code كـ skill واحدة وتستخدم حقل name في الـ frontmatter كاسم للاستدعاء. استخدم هيكل skills/ للـ plugins التي قد تكبر لأكثر من skill.

الـ manifest يعرّف الـ plugin والـ metadata الخاصة به:

.claude-plugin/plugin.json
{
  "name": "pr-review",
  "description": "Complete PR review workflow with security and test coverage checks",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  },
  "repository": "https://github.com/you/pr-review",
  "license": "MIT"
}
ⓘأوامر مميّزة بالـ namespace
أوامر الـ plugin والـ skills التي يوفّرها تكون namespaced بصيغة plugin-name:command-name لتجنب التعارض مع إعدادات المشروع. شغّلها بالاسم الكامل مثل /pr-review:check-security.

ميزات الـ Manifest

الـ manifest يدعم عدة خصائص قوية لضبط سلوك الـ plugin. خاصية userConfig تعلن عن خيارات قابلة للتخصيص من المستخدم. الحقول المعلّمة بـ sensitive: true تُخزَّن في الـ system keychain بدلًا من ملفات الإعدادات النصية:

.claude-plugin/plugin.json — userConfig
{
  "name": "my-plugin",
  "version": "1.0.0",
  "userConfig": {
    "apiKey": {
      "description": "API key for the integration",
      "sensitive": true
    },
    "region": {
      "description": "Deployment region",
      "default": "us-east-1"
    }
  }
}

الـ plugins تحصل على مجلد بيانات دائم عبر ${CLAUDE_PLUGIN_DATA} (من الإصدار v2.1.78). هذا يبقى موجودًا بين الجلسات، فمناسب للـ caches وملفات الحالة والـ databases. استخدم ${CLAUDE_PLUGIN_ROOT} للإشارة إلى مسارات نسبية من مجلد تثبيت الـ plugin — ضروري للـ hooks وإعدادات MCP:

.claude-plugin/plugin.json — hooks مع CLAUDE_PLUGIN_ROOT
{
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node ${CLAUDE_PLUGIN_ROOT}/bin/audit.js"
          }
        ]
      }
    ]
  }
}

مراقبات الـ Plugin

المراقبات تحتاج Claude Code v2.1.105 أو أحدث. عرّفها تحت مفتاح experimental في الـ manifest (experimental.monitors) — الشكل top-level لا يزال يعمل لكن claude plugin validate سيعرض تحذيرًا، وإصدار قادم سيجعل الشكل المتداخل إلزاميًا. مفتاح monitors في الـ manifest يربط الـ plugin بأداة Monitor. ضع JSON file (أو الإعداد inline) وتتفعّل المراقبات الخلفية تلقائيًا عند تفعيل الـ plugin في بداية الجلسة، أو عند استدعاء أي skill من الـ plugin:

.claude-plugin/plugin.json — monitors
{
  "name": "ci-watcher",
  "version": "1.0.0",
  "experimental": {
    "monitors": "./monitors.json"
  }
}
▲مسار monitors مخصص
عندما يحدد الـ manifest مسار monitors مخصصًا، الموقع الافتراضي monitors/monitors.json لا يُفحص بعد الآن — أضفه صراحةً إذا أردت تحميله بجانب ملفك المخصص.

دعم LSP

دعم LSP يضيف تكاملًا مع language server protocol في الوقت الفعلي. ضع ملف .lsp.json في root الـ plugin لضبط language servers توفّر تشخيصات فورية و go-to-definition وبحث عن symbols أثناء التعديل:

.lsp.json
{
  "typescript": {
    "command": "typescript-language-server",
    "args": ["--stdio"],
    "extensionToLanguage": {
      ".ts": "typescript",
      ".tsx": "typescriptreact"
    }
  }
}

التوزيع والتطوير

جرّب الـ plugin محليًا بـ flag الـ --plugin-dir قبل التوزيع. يحمّل الـ plugin للجلسة الحالية فقط — بدون تثبيت:

bash
claude --plugin-dir ./my-plugin
# اختبر عدة plugins في نفس الوقت:
claude --plugin-dir ./my-plugin --plugin-dir ./another-plugin

للـ plugins المستضافة كأرشيفات .zip، يحمّلها --plugin-url ويثبّتها للجلسة الحالية فقط بدون تثبيت دائم. كرّر الـ flag لعدة plugins:

bash
claude --plugin-url https://example.com/my-plugin.zip
claude --plugin-url https://example.com/a.zip --plugin-url https://example.com/b.zip
▲أمان — أرشيفات خارجية
استخدم --plugin-url فقط مع روابط موثوقة — تحميل أرشيفات من مصادر خارجية يشغّل كودًا تابعًا لطرف ثالث على جهازك.

استخدم /reload-plugins لإعادة تحميل ملفات الـ plugin أثناء التطوير دون إعادة تشغيل الجلسة. يعيد قراءة جميع الـ manifests والـ skills والـ agents والـ hooks وإعدادات MCP فورًا.

شاشات /plugin بتاعت Discover و Browse تعرض inventory كامل لكل ما سيثبّته الـ plugin قبل اتخاذ القرار — commands وagents وskills وhooks وأي MCP أو LSP servers يشحنها. هذا يجعل تقييم plugin من الـ marketplace قرارًا في شاشة واحدة.

التثبيت من الـ Marketplace

توزيع الـ plugins يتبع نموذج marketplace. الـ marketplace الرسمي من Anthropic هو claude-plugins-official. أضف marketplaces إضافية بـ /plugin marketplace add owner/repo-name. ثبّت plugins بـ /plugin install plugin-name أو claude plugin install plugin-name@marketplace:

bash
# ثبّت من الـ marketplace الرسمي
/plugin install pr-review

# ثبّت من GitHub
/plugin install github:username/my-plugin

# ثبّت من مسار محلي (للاختبار)
/plugin install ./path/to/plugin

عندما يكون مصدر الـ plugin اختصار GitHub بصيغة owner/repo، تستنسخه Claude Code افتراضيًا عبر SSH. هذا يتعطل في بيئات CI والـ containers أو أي بيئة بلا SSH key مضبوط لـ github.com. اضبط CLAUDE_CODE_PLUGIN_PREFER_HTTPS=1 لإجبار الاستنساخ عبر HTTPS:

bash
# اجبر HTTPS في الـ CI عند تثبيت plugins
CLAUDE_CODE_PLUGIN_PREFER_HTTPS=1 claude plugin install owner/repo

عرض تفاعلي — تثبيت plugin واستدعاء skill بالـ namespace:

claude — plugin install
>

التفعيل والتعطيل والتنظيف

claude plugin enable و claude plugin disable يشغّلان ويوقفان plugin مثبّت دون حذفه. الاثنان يقبلان اسم <plugin> (أو <plugin>@<marketplace> للتمييز) و --scope بقيمة user أو project أو local (الافتراضي: user). التعطيل على نطاق project يكتب الخيار في .claude/settings.json ليستلمه الفريق بأكمله؛ نطاق user يبقيه شخصيًا:

bash
# شخصي: عطّل plugin مزعج لك فقط
claude plugin disable formatter@anthropics/claude-plugins

# فريق: أبقِ الـ plugin في الإعدادات لكن عطّله على مستوى المشروع
claude plugin disable formatter --scope project

# فعّله لاحقًا دون المساس بالتثبيت أو الإصدار
claude plugin enable formatter --scope project

claude plugin prune (جديد في v2.1.121، له الاسم البديل autoremove) يحذف الـ dependencies التي ثُبِّتت تلقائيًا ولا يحتاجها أي plugin مثبّت آخر — الـ plugins التي ثبّتها بنفسك لا تُمسّ. لإلغاء تثبيت plugin وتنظيف dependencies بتاعته دفعة واحدة:

bash
# ألغِ تثبيت plugin ونظّف الـ auto-installs اليتيمة
claude plugin uninstall my-plugin --prune

# أو نظّف فقط (أبقِ جميع الـ plugins التي ثبّتها مباشرة)
claude plugin prune

عرض تفاعلي — تعطيل على مستوى المشروع، إعادة التفعيل، ثم تنظيف اليتيمة:

claude — plugin manage
>

أوامر دورة الحياة

نظرة سريعة على أوامر دورة حياة الـ plugin المدمجة:

الأمرالوظيفة
claude plugin listعرض جميع الـ plugins المثبّتة
claude plugin enable <name>تفعيل plugin مثبّت
claude plugin disable <name>تعطيل دون إلغاء التثبيت
claude plugin uninstall <name>حذف plugin نهائيًا
claude plugin validateالتحقق من الـ manifest والتحذير من الحقول المهجورة
claude plugin prune / autoremoveحذف الـ dependencies اليتيمة المثبّتة تلقائيًا
claude plugin details <name>عرض skills وagents وhooks وMCP/LSP وتكلفة الـ tokens
claude plugin tagإنشاء git tag للإصدار (v2.1.118+، مع التحقق من الإصدار)

الـ Plugins المضمّنة في الإعدادات

نمط الـ inline plugin (source: 'settings' من الإصدار v2.1.80) يتيح تعريف plugin مباشرة في ملف إعدادات دون الحاجة لـ repository منفصل. مفيد للأدوات الداخلية الصغيرة التي لا تستحق إنشاء repo كامل:

.claude/settings.json — inline plugin
{
  "pluginMarketplaces": [
    {
      "name": "internal-tools",
      "source": "settings",
      "plugins": [
        {
          "name": "code-standards",
          "source": "./local-plugins/code-standards"
        }
      ]
    }
  ]
}

ضوابط المؤسسات

في بيئات المؤسسات، يتحكم ملف managed-mcp.json في أي MCP servers يمكن للـ plugins استخدامها. إعدادات enabledPlugins وextraKnownMarketplaces وstrictKnownMarketplaces وblockedMarketplaces في الـ managed policy تتحكم في أي plugins و marketplaces مسموح بها على مستوى المؤسسة.

▲قيود subagents الـ plugins
الـ subagents الخاصة بالـ plugins لها frontmatter محدود — لا يمكنها تعريف hooks أو mcpServers أو permissionMode لمنع privilege escalation.

اختبر نفسك — الـ Plugins

1. ما هو الملف الوحيد المطلوب داخل مجلد الـ plugin؟

2. كيف يتم تجنب تعارض أوامر الـ plugins مع إعدادات المشروع؟

3. ماذا يفعل الأمر `claude plugin prune` (المعروف أيضًا بـ autoremove)؟

4. ما هو متغير البيئة الذي يجبر الاستنساخ عبر HTTPS بدلًا من SSH عند تثبيت plugin من GitHub؟
